# Как работают сценарии Lola

Сценарии помогают Lola реагировать на действия пользователя по заранее заданным правилам. Например:

- поприветствовать человека после регистрации;
- показать подсказку, когда он открыл нужную страницу;
- подождать первый депозит и продолжить по одной из двух веток;
- напомнить о незавершённом действии;
- дождаться, когда человек снова появится онлайн, и только тогда показать сообщение.

Сценарий не принимает решения сам и не придумывает следующий шаг. Он выполняет опубликованные настройки в том порядке, который задал администратор.

Эта страница объясняет всю модель простыми словами: что такое событие, как строятся условия, чем аудитория отличается от условий запуска, как работают ожидания, публикация и история версий.

Материал можно читать по порядку или использовать как справочник. Сначала разобраны события и правила запуска, затем действия и ожидания, а после них работа с черновиком, публикацией и историей.

## Коротко: из чего состоит сценарий

У любого сценария есть пять основных частей.

1. Trigger: событие, с которого начинается проверка сценария.
2. Audience: кому предназначен сценарий.
3. Eligibility: подходит ли ситуация прямо сейчас.
4. Actions: что Lola должна сделать.
5. Delivery Policy: что делать, если для выполнения действия пользователь должен быть онлайн, но сейчас недоступен.

Внутри последовательности действий можно добавлять ожидания, вопросы и развилки. Например, сценарий может подождать депозит два дня, а затем выбрать ветку "депозит сделан" или "срок истёк".

Простой пример:

```text
Событие: пользователь зарегистрировался
Аудитория: язык пользователя русский
Условие: приветственный сценарий ещё не запускался
Действие: отправить приветственное сообщение
Доставка: если пользователь offline, ждать его не больше одного дня
```

### Project и End User

Project — отдельный продукт или подключённая среда со своими пользователями, событиями и сценариями. Данные разных Project не смешиваются.

End User — конкретный пользователь внутри Project. Один и тот же внешний идентификатор в двух разных Project не превращает их в одного общего пользователя.

## Событие: зафиксированный факт

Event, или событие, означает, что с конкретным пользователем что-то произошло. Событие всегда относится к одному Project и одному End User.

Примеры событий:

- пользователь зарегистрировался;
- открыл страницу акций;
- успешно пополнил баланс;
- нажал кнопку;
- начал новый визит;
- появился онлайн.

Событие нельзя изменить задним числом. Если оно принято системой, оно остаётся в истории как факт с точным временем и данными, которые были известны в тот момент.

Принятые события можно смотреть в Event Logs. Это журнал, по которому администратор проверяет, что действительно пришло, когда это произошло и какие сценарии могли отреагировать. Реальную запись из Event Logs также можно выбрать для Preview.

### Откуда приходят события

События могут создавать разные части продукта.

- Frontend сообщает о действиях в интерфейсе: открытии страницы, нажатии кнопки, выборе элемента.
- Backend сообщает о доверенных операциях: регистрации, платеже, изменении статуса заказа.
- Lola создаёт системные события: начало нового визита, первый активный день, переход пользователя в online или offline.

Для каждого события заранее указано, откуда его разрешено отправлять. Например, финансовое событие нельзя принять от браузера, если оно объявлено серверным. Это защищает сценарии от поддельных фактов.

Если отправитель повторит одно и то же событие из-за сетевого сбоя, система распознает повтор и не запустит сценарий второй раз.

## Event Definition: описание события

Само событие является фактом. Event Definition описывает, как этот факт устроен.

Возьмём событие "Успешный депозит". Его описание может содержать такие поля:

```text
Сумма: число
Валюта: EUR или USD
Способ оплаты: текст
```

По этому описанию система понимает:

- какие данные обязательны;
- какие значения допустимы;
- какие поля можно использовать в условиях;
- какие числа можно суммировать;
- какие данные нужно скрывать в объяснениях;
- можно ли отправлять событие из frontend.

Неправильно составленное событие не попадёт в историю и не запустит сценарии.

### Почему у описания события есть версии

Описание события со временем меняется. В него могут добавить поле, переименовать свойство или уточнить смысл суммы. Опубликованное описание не переписывается. Вместо этого создаётся следующая версия.

Это нужно для предсказуемости. Сценарий, опубликованный с одной версией события, продолжает понимать её так же, как в день публикации. Новые сценарии могут использовать новую версию.

В истории Event Definition можно увидеть:

- текущую версию;
- предыдущие версии;
- какие сценарии всё ещё зависят от старой версии;
- совместимо ли предполагаемое изменение с опубликованными сценариями.

У системных событий Lola стоит признак read-only. Их можно выбирать в сценариях, но нельзя переименовывать, удалять или менять их поля.

## Данные события

Дополнительные данные события называются payload. У события открытия страницы это может быть код страницы, у депозита — сумма и валюта.

Администратор не вводит технический путь к данным вручную. Редактор показывает только поля, разрешённые Event Definition. Благодаря этому условие не сломается из-за опечатки вроде `ammount` вместо `amount`.

Поддерживаются обычные типы данных:

- текст;
- число;
- да/нет;
- дата и время;
- значение из заранее заданного списка.

Вложенные поля тоже могут быть доступны, если система умеет безопасно использовать их в сценариях. Если поле пока нельзя применять, редактор должен показать причину, а не предлагать неработающую настройку.

### Денежные суммы

Для денег отдельно задаются единица хранения и способ показа. Например, backend может хранить `50000` минимальных единиц, а CMS показывать `500.00`.

Валюта всегда учитывается явно. Система не складывает EUR и USD в одну сумму. Для денежной агрегации нужно выбрать одну валюту, например:

```text
Сумма успешных депозитов в EUR за последние 30 дней не меньше 500 EUR
```

Если описание денежного поля не содержит однозначных правил отображения, его нельзя опубликовать для использования в сценариях.

## Trigger: что начинает сценарий

Trigger — это новое событие, после которого система проверяет сценарий.

Trigger ещё не означает, что сценарий обязательно запустится. Он только говорит: "Произошло нужное событие, теперь проверь Audience, Eligibility и ограничения запуска".

Например:

```text
Trigger: page.opened
Eligibility: открыта именно страница promotions
```

Событие `page.opened` может происходить на разных страницах. Условие уточняет, какой случай нужен сценарию.

Один Event может быть Trigger для нескольких сценариев. Каждый из них проверяется независимо.

## Audience: для кого предназначен сценарий

Audience описывает группу пользователей, для которой создан сценарий. Она смотрит на текущее состояние End User, а не на детали только что пришедшего события.

В Audience можно использовать, например:

- язык или locale;
- типизированные атрибуты пользователя;
- принадлежность к Segment;
- сочетание нескольких правил.

Пример:

```text
Язык пользователя = русский
И
Пользователь входит в Segment "Новые пользователи"
```

Segment — это переиспользуемое правило аудитории. Его удобно применять в нескольких сценариях. У Segment тоже есть версии: изменение правила не переписывает уже опубликованную версию.

### Audience и Eligibility — не одно и то же

Эти понятия легко перепутать.

Audience отвечает на вопрос: "Этот пользователь относится к нужной группе?"

Eligibility отвечает на вопрос: "Подходит ли текущая ситуация для запуска?"

Пример:

```text
Audience: русскоязычные новые пользователи
Eligibility: регистрация произошла сегодня и приветствие ещё не запускалось
```

Результаты сохраняются отдельно. Поэтому в истории запуска видно, пользователь не входил в нужную аудиторию или не выполнилось поведенческое условие.

## Eligibility: когда сценарий разрешено запустить

Eligibility проверяется после Trigger. Условия могут смотреть на текущее событие, историю событий и рассчитанную активность пользователя.

### Условия по текущему событию

Такое условие проверяет payload Trigger.

Примеры:

```text
Страница = promotions
Сумма текущего депозита больше 100 EUR
Способ оплаты входит в список card, bank_transfer
Поле campaignCode заполнено
```

Доступные сравнения зависят от типа поля. Для текста и числа набор операций будет разным. Редактор получает допустимые варианты из каталога и не должен придумывать их самостоятельно.

### Условия по истории

Историческое условие считает события пользователя за ограниченный период.

Примеры:

```text
Был хотя бы один успешный депозит за последние 7 дней
Количество открытий страницы за последние 24 часа больше 3
Сумма депозитов в EUR за 30 дней не меньше 500 EUR
Время первого события в периоде раньше заданной даты
```

Для истории можно использовать следующие способы подсчёта:

- `exists`: было ли хотя бы одно событие;
- `count`: сколько событий произошло;
- `sum`: сумма числового поля;
- `min` и `max`: минимальное или максимальное значение;
- `first` и `last`: время первого или последнего события.

Исторический период всегда ограничен. Сценарий не может незаметно просматривать всю историю всех пользователей. Это защищает систему от слишком тяжёлых правил.

### Составные условия

Условия можно объединять:

- ALL: должны выполниться все дочерние условия;
- ANY: достаточно одного;
- NOT: результат вложенного условия меняется на противоположный.

Пример:

```text
ALL
  Валюта текущего события = EUR
  ANY
    Сумма депозитов за 30 дней >= 500 EUR
    Пользователь входит в Segment "VIP"
  NOT
    Было событие self_excluded
```

Вложенность и количество условий ограничены. Если правило становится слишком большим или дорогим, проверка перед публикацией объяснит, что нужно упростить.

## Ограничения самого сценария

Кроме Audience и Eligibility, у сценария есть общие ограничения запуска.

- Период активности: даты, когда сценарий разрешено запускать.
- Cooldown: пауза между повторными запусками для одного пользователя.
- Максимальное число запусков на пользователя.
- Priority: порядок обработки, если несколько сценариев подходят одновременно.
- Статус сценария.

Статусы означают следующее:

- Draft: сценарий ещё не опубликован и не запускается;
- Active: сценарий может запускаться;
- Paused: новые запуски временно остановлены;
- Archived: сценарий убран из активной работы и хранится для истории.

## Действия и схема прохождения

После успешных проверок создаётся Scenario Run. Он проходит по опубликованной схеме действий.

В простом сценарии шаги идут один за другим:

```text
Показать Lola
Отправить сообщение
Показать кнопку
Завершить сценарий
```

В более сложном сценарии появляются ветки:

```text
Задать вопрос
  Ответ "Да" -> открыть страницу
  Ответ "Нет" -> завершить сценарий
  Нет ответа вовремя -> отправить напоминание
```

У каждого шага есть стабильное место в схеме. При публикации система проверяет, что все переходы ведут в существующие шаги, нет дубликатов и недостижимых частей сценария.

### Какие действия доступны

Набор действий определяется backend и может расширяться. Сейчас сценарий умеет:

- показать или скрыть ассистента;
- открыть или закрыть чат;
- написать сообщение в чат;
- озвучить текст;
- начать голосовой диалог;
- проиграть анимацию;
- подсветить элемент интерфейса или убрать подсветку;
- показать CTA-кнопку;
- открыть зарегистрированную страницу или модальное окно;
- подождать заданное время;
- ждать Goal до Deadline;
- задать вопрос с вариантами ответа;
- выбрать ветку по условию;
- записать аналитическое событие шага;
- завершить Scenario Run.

CMS показывает только опубликованные и разрешённые действия. Если действие отключено на backend, его нельзя использовать в новом сценарии.

### Зарегистрированные страницы, окна и элементы

Сценарий не передаёт в frontend произвольную ссылку или CSS-селектор. Страницы, модальные окна, кнопки и другие цели заранее регистрируются в Project.

Администратор выбирает понятное название из списка. Backend сохраняет безопасный идентификатор. Это снижает риск открыть несуществующую страницу или выполнить неизвестную команду.

## Шаблоны в сообщениях

В тексты можно подставлять данные текущего запуска.

Пример:

```text
Привет, {{ user.profile.name }}!
Ваш депозит: {{ event.payload.amount }} {{ event.payload.currency }}.
```

Шаблоны поддерживаются в сообщениях чата, озвучиваемом тексте, первой реплике голосового диалога и вопросах с вариантами ответа.

Доступны только разрешённые источники:

- `event`: данные Trigger;
- `user`: данные End User;
- `project`: данные Project;
- `scenario`: данные опубликованного сценария.

Шаблон не может обратиться к системным переменным, секретам или произвольному коду.

Если поле может отсутствовать, нужно задать запасной текст:

```text
Привет, {{ user.profile.name | default: "друг" }}!
```

Без значения по умолчанию сценарий с необязательной переменной не пройдёт проверку. Если обязательное значение неожиданно отсутствует во время запуска, шаг завершается ошибкой, а не отправляет пользователю текст с пустым местом.

## Обычное ожидание времени

Действие "Подождать" приостанавливает Scenario Run на заданный срок, после чего он продолжает следующий шаг.

Ожидание хранится надёжно. Перезапуск приложения или обновление сервера не сбрасывает таймер и не начинает его заново.

Такое ожидание подходит, когда дальнейший путь не зависит от действий пользователя:

```text
Отправить сообщение
Подождать 15 минут
Показать вторую подсказку
```

Если нужно дождаться конкретного результата, используется Goal.

## Goal и Deadline

Goal — это будущий факт, которого ждёт уже запущенный сценарий. Deadline — крайний срок ожидания.

Пример:

```text
Goal: количество успешных депозитов после регистрации >= 1
Deadline: через 2 дня после Trigger
```

У ожидания всегда две ветки:

- onGoal: цель достигнута вовремя;
- onTimeout: срок закончился раньше, чем выполнилась цель.

Полная схема может выглядеть так:

```text
Пользователь зарегистрировался
Ждать первый успешный депозит не больше 2 дней
  Депозит сделан -> завершить сценарий без напоминания
  Срок истёк -> отправить напоминание
```

### Что можно считать в Goal

Goal работает с одним типом события и поддерживает:

- `count`: количество подходящих событий;
- `sum`: сумму выбранного числового поля.

События можно отфильтровать по payload. Например, считать только успешные депозиты в EUR.

### Точная граница срока

Событие засчитывается, если произошло до Deadline. Событие ровно в момент Deadline или позже относится к опоздавшим. В таком случае выбирается timeout-ветка.

Event и Deadline могут прийти почти одновременно. Система всё равно выбирает только одного победителя. Обе ветки одного ожидания никогда не должны продолжиться вместе.

Повторная доставка того же события тоже не увеличивает Goal второй раз.

### Что происходит после перезапуска

Незавершённый Goal хранится отдельно от процесса приложения. После перезапуска ожидание остаётся активным, сохраняет накопленный результат и завершается по тем же правилам.

Завершённые ожидания не лежат в рабочей очереди бесконечно. Итог остаётся в истории Scenario Run: цель выполнена, истёк срок или произошла ошибка.

## Вопрос с вариантами ответа

Действие "Задать вопрос" отправляет сообщение и набор кнопок. Для каждого ответа заранее выбирается следующая ветка.

Также задаются:

- срок ожидания ответа;
- ветка при отсутствии ответа;
- необязательные напоминания во время ожидания.

Первый допустимый ответ завершает ожидание. Повторный ответ или timeout после принятого ответа не запускают другую ветку.

Напоминания состоят из обычных разрешённых действий. Их настройки проверяются так же строго, как основные шаги сценария.

## Развилка по условию

Действие "Условие" проверяет ветки сверху вниз и выбирает первую совпавшую. Если ни одна ветка не подошла, используется обязательная fallback-ветка.

Порядок веток имеет значение. Более узкие и важные условия стоит размещать выше общих.

Развилка внутри Scenario Run отличается от Eligibility. Eligibility решает, начинать ли Run вообще. Развилка выбирает путь внутри уже запущенного Run.

## Delivery Policy: что делать, когда пользователь offline

Некоторые действия требуют активного frontend: открыть страницу, показать CTA, запустить анимацию. Если пользователь offline, такое действие невозможно выполнить сразу.

Delivery Policy задаётся для опубликованного сценария и определяет поведение таких шагов.

### Выполнить немедленно

Система пытается выполнить действие сейчас. Если пользователя нет online, шаг завершается ошибкой. Этот вариант подходит для действий, которые имеют смысл только в текущем визите.

### Пропустить, если offline

Если пользователь недоступен, действие считается пропущенным, а Scenario Run продолжает следующую ветку. Ошибки доставки нет.

### Завершить ошибкой, если offline

Если пользователя нет online, шаг и Scenario Run фиксируют явную ошибку `USER_OFFLINE`. Этот вариант нужен, когда без действия дальнейшее выполнение теряет смысл.

### Дождаться online

Scenario Run приостанавливается и ждёт, пока пользователь появится online. У ожидания обязательно есть конечный срок, не больше семи дней.

Когда пользователь возвращается, действие возобновляется. При желании перед доставкой можно ещё раз проверить Eligibility и Audience. Это полезно, если за время ожидания ситуация могла измениться.

Если срок ожидания online закончился, шаг получает результат Delivery timeout. Он не остаётся ждать бесконечно.

### Goal и ожидание online решают разные задачи

Goal ждёт бизнес-результат, например депозит.

Delivery Policy ждёт техническую возможность показать готовое действие.

Сначала Goal должен завершиться успехом или timeout. После этого действие выбранной ветки при необходимости может отдельно ждать online.

## Presence, Visit и Activity Day

Система различает техническое подключение и реальную активность человека.

Presence показывает, доступен ли пользователь сейчас для интерактивной доставки. Короткий обрыв соединения не обязательно означает, что человек действительно ушёл: система учитывает небольшой период на переподключение.

Visit начинается после новой осмысленной активности, если с предыдущей активности прошло больше настроенного интервала. Открытие второй вкладки, heartbeat или автоматическое продление сессии не создают новый Visit.

Activity Day означает, что пользователь был активен в конкретную календарную дату Project. Несколько вкладок и повторные подключения всё равно дают один Activity Day.

Системные события Lola:

- `lola.visit_started`: начался новый визит после периода неактивности;
- `lola.activity_day_started`: первая осмысленная активность в локальный календарный день;
- `lola.became_online`: устойчивый переход из offline в online;
- `lola.became_offline`: устойчивый переход из online в offline.

На основе Activity Day можно строить правила вроде "пользователь активен два дня подряд". Для этого не нужно считать подключения или технические сессии.

Часовой пояс и интервал, отделяющий один Visit от другого, настраиваются для Project.

## Что происходит при запуске

Когда приходит Event, система действует в таком порядке.

1. Проверяет, существует ли подходящее Event Definition и разрешён ли источник.
2. Проверяет структуру и значения payload.
3. Сохраняет Event в истории.
4. Находит активные сценарии с таким Trigger.
5. Проверяет период активности, cooldown и лимит запусков.
6. Отдельно вычисляет Audience и Eligibility.
7. Для подходящего сценария создаёт Scenario Run на точной опубликованной версии.
8. Выполняет действия, пока Run не завершится, не остановится на ожидании или не встретит ошибку.

Каждый новый Scenario Run получает собственный снимок решения. Изменение пользователя или новая публикация сценария не переписывают уже сохранённый старт.

Если у пользователя уже есть незавершённый Run того же сценария, новый запуск отменяет предыдущий. В истории будет видно, какой Run был отменён и какой его заменил.

## Что такое Scenario Run

Scenario Run — одно прохождение опубликованного сценария одним End User после одного Trigger.

Run хранит:

- точную версию сценария;
- Trigger и время запуска;
- результаты Audience и Eligibility;
- пройденные шаги;
- активные и завершённые ожидания;
- выбранные ветки;
- результат доставки;
- итоговый статус.

Возможные итоговые состояния:

- Running: выполнение продолжается или чего-то ждёт;
- Completed: сценарий успешно завершён;
- Failed: один из обязательных шагов завершился ошибкой;
- Skipped: запуск был пропущен по правилу;
- Cancelled: выполнение отменено, например новым запуском;
- Expired: закончился допустимый срок выполнения.

У шагов есть собственные состояния: ожидает запуска, выполняется, ждёт время, ответ, подтверждение frontend или online-доставку, завершён, пропущен либо завершён ошибкой.

## Чат и разговор

Сценарий может создать новый разговор или использовать текущий активный разговор пользователя. Это задаётся Conversation Policy.

- Create new создаёт отдельный разговор для нового Scenario Run.
- Reuse active использует текущий открытый разговор, если он есть, иначе создаёт новый.

Пустой разговор не создаётся заранее только потому, что Scenario Run ждёт два дня. Он появляется перед первым действием, которому действительно нужен чат или голос.

## Черновик

Черновик — редактируемая версия сценария. Он сохраняется отдельно от опубликованного сценария и никогда не исполняется.

В черновике хранятся:

- Rule и Audience в форме, удобной для редактора;
- Delivery Policy;
- полная схема действий и переходов;
- версия каталога событий, по которой собраны условия;
- опубликованная версия, от которой началось редактирование.

Черновик сохраняется на backend. Поэтому после перезагрузки страницы его можно восстановить и продолжить работу.

### Защита от случайной перезаписи

У черновика есть номер версии. При сохранении CMS сообщает, какую версию открывал пользователь.

Если другой администратор уже сохранил изменения, backend не перезапишет их молча. CMS получит конфликт и предложит перечитать свежий черновик.

Такая же проверка действует для опубликованной версии. Нельзя безопасно опубликовать черновик, собранный поверх уже устаревшего состояния сценария.

## Проверка перед публикацией

Проверка не меняет сценарий и не запускает его. Она собирает все найденные проблемы, чтобы CMS могла показать их рядом с нужными полями.

Проверяются:

- Trigger и его текущая версия;
- Audience и Eligibility;
- доступность полей, сравнений и агрегатов;
- денежные суммы и валюта;
- Delivery Policy;
- настройки каждого действия;
- шаблоны и значения по умолчанию;
- Goal, Deadline и обе ветки;
- вопросы, напоминания и timeout-ветки;
- существование всех переходов;
- отсутствие циклов назад и недостижимых шагов;
- стоимость исторических условий.

Если каталог событий изменился после открытия редактора, проверка попросит обновить данные. Это лучше, чем опубликовать сценарий с устаревшим полем.

## Preview

Preview помогает проверить правило на реальном событии из Event Logs того же Project.

Администратор выбирает запись события. Backend сам получает связанного пользователя, payload и время. Нельзя передать произвольные данные и выдать их за реальное событие.

Preview показывает:

- совпало правило или нет;
- какие части условия выполнились;
- фактическое значение;
- сколько событий попало в расчёт;
- какое временное окно использовалось;
- какие данные были скрыты.

Для Goal есть отдельный preview. Он считает `count` или `sum` по точной версии события, показывает Deadline и объясняет, достигнут ли порог.

Чувствительные значения не отображаются. Вместо суммы или другого приватного факта CMS увидит отметку "Скрыто". При этом общий результат проверки остаётся понятным.

## Публикация

Публикация превращает проверенный черновик в неизменяемую Scenario Revision.

В опубликованной версии фиксируются:

- Trigger;
- Rule и Audience;
- схема действий;
- Goal и переходы;
- Delivery Policy;
- зависимости от точных версий Event Definition и Segment;
- ограничения запуска;
- автор и время публикации.

Backend также готовит внутреннее исполняемое представление. Редактор продолжает работать с понятным исходным документом, а runtime получает проверенный снимок без свободных формул и произвольных путей.

Публикация и переключение сценария на новую версию происходят вместе. Не бывает состояния, в котором половина данных уже обновлена, а половина ещё старая.

Если публикуется сохранённая версия черновика, backend берёт Rule, Audience, Delivery Policy и схему именно из неё. После успешной публикации этот черновик удаляется. Более новая чужая версия черновика случайно удалена не будет.

## Что происходит с уже запущенными сценариями

Новая публикация влияет только на будущие Scenario Runs.

Уже запущенный Run продолжает точную версию, с которой стартовал. Его Goal, Deadline, тексты, ветки и Delivery Policy не меняются посередине пути.

Это правило особенно важно для долгих ожиданий. Пользователь, который начал двухдневный сценарий вчера, не должен внезапно перейти на новые условия, опубликованные сегодня.

## История версий и откат

Каждая публикация создаёт новую запись в истории. Предыдущие версии не редактируются и не удаляются из-под активных Runs.

В истории видно:

- номер и идентификатор версии;
- время публикации;
- автора;
- является ли версия текущей;
- можно ли открыть её как редактируемый исходник.

Очень старые версии, созданные до появления authoring read model, могут содержать только исполняемый снимок. Система не пытается угадать исходные поля. Для такой версии CMS показывает, что безопасное редактирование недоступно.

Rollback не переписывает прошлое. Backend берёт выбранную версию и публикует её копию как новую текущую версию. Поэтому в истории остаётся понятная цепочка действий.

## Объяснение запуска

Для каждого Scenario Run доступно безопасное объяснение.

Оно показывает:

- какую версию сценария выполнил Run;
- какой Event был Trigger;
- результаты Audience и Eligibility;
- какие ветки были выбраны;
- чем завершился Goal: Event или Deadline;
- ждала ли доставка online;
- статусы шагов и команд;
- последовательность важных переходов.

Объяснение не раскрывает внутренние служебные данные, секреты, полный профиль пользователя или raw payload. Чувствительные значения заменяются отметкой о скрытии.

## Три полных примера

### Приветствие после регистрации

```text
Trigger
  Пользователь зарегистрировался

Audience
  Locale = ru

Eligibility
  Приветствие ещё не запускалось

Actions
  Показать Lola
  Написать "Привет, {{ user.profile.name | default: "друг" }}!"
  Показать CTA "Посмотреть возможности"

Delivery
  Выполнить немедленно
```

Если пользователь online и все проверки пройдены, действия выполнятся по порядку. Если интерактивный шаг требует frontend, а пользователь уже ушёл, немедленная доставка завершит этот шаг ошибкой.

### Первый депозит после регистрации

```text
Trigger
  Пользователь зарегистрировался

Audience
  Новые пользователи

Goal
  count(deposit.succeeded, currency = EUR) >= 1
  Deadline: 2 дня

onGoal
  Завершить сценарий

onTimeout
  Написать напоминание

Delivery
  Ждать online не больше 1 дня
  Перед доставкой перепроверить условия
```

Если депозит произойдёт до Deadline, сценарий выберет onGoal. Если нет, выберет onTimeout. Напоминание может отдельно ждать появления пользователя online.

### Подсказка на странице акций

```text
Trigger
  Открыта страница

Eligibility
  pageCode = promotions
  И пользователь ещё не нажимал promo_cta за последние 7 дней

Actions
  Показать Lola
  Подсветить кнопку promo_cta
  Показать CTA "Получить бонус"

Delivery
  Пропустить интерактивное действие, если пользователь offline
```

Сценарий сработает только для нужной страницы и не станет повторять подсказку пользователю, который недавно уже нажал кнопку.

## Как собрать сценарий без ошибок

Перед публикацией полезно пройти короткий список.

1. Выбрать Trigger и проверить, что это именно событие начала, а не будущий Goal.
2. Отделить Audience от Eligibility.
3. Для исторических условий выбрать понятный ограниченный период.
4. Для денег указать одну валюту и проверить единицы отображения.
5. Убедиться, что каждая развилка имеет fallback или timeout-ветку.
6. Для Goal задать конечный Deadline и обе ветки.
7. Для ожидания online указать конечный срок.
8. Добавить значения по умолчанию для необязательных переменных в шаблонах.
9. Запустить полную проверку и Preview на реальном Event.
10. Перед публикацией просмотреть изменения относительно текущей версии.

## Частые вопросы

### Сценарий начинается сразу после любого Event?

Нет. Event должен совпасть с Trigger. После этого ещё проверяются статус сценария, период активности, cooldown, лимит запусков, Audience и Eligibility.

### Можно изменить опубликованный сценарий?

Опубликованная версия неизменяема. Изменения вносятся в черновик и публикуются как новая версия.

### Изменится ли уже запущенный Run после новой публикации?

Нет. Он продолжит версию, с которой начался.

### Что будет после перезагрузки редактора?

Сохранённый backend-черновик восстановится вместе с номером версии и текущей опубликованной основой. Его можно продолжить редактировать.

### Что будет, если два администратора редактируют одновременно?

Backend не даст второму сохранению молча затереть первое. CMS получит конфликт версии и должна предложить загрузить свежие изменения.

### Почему Trigger нельзя заменить Goal?

Trigger начинает новый Run. Goal относится к уже начатому Run и ждёт будущий результат до Deadline.

### Почему Audience нельзя смешивать с Eligibility?

Они отвечают на разные вопросы и могут меняться по разным причинам. Раздельный результат делает запуск понятным и позволяет безопасно перепроверить состояние перед отложенной доставкой.

### Может ли ожидание длиться бесконечно?

Нет. У Goal есть Deadline, у ожидания online есть expiry, у вопросов есть timeout. Обычный таймер тоже имеет максимальную длительность.

### Что будет, если Event и Deadline произойдут одновременно?

Продолжится только одна ветка. Победитель фиксируется в истории, повторная обработка ничего не меняет.

### Можно вставить в условие SQL, JavaScript или собственную формулу?

Нет. Сценарии собираются только из типизированных полей и операций, разрешённых backend-каталогом.

### Почему в Preview иногда написано "Скрыто"?

Поле помечено как чувствительное. Система показывает результат условия, но не раскрывает само значение.

### Чем Visit отличается от online?

Online означает, что действие можно доставить сейчас. Visit означает новый период осмысленной активности после паузы. Короткое переподключение может восстановить online, но не создаёт новый Visit.

## Главное, что стоит запомнить

Event сообщает, что произошло. Trigger решает, когда начать проверку. Audience отвечает, подходит ли пользователь. Eligibility проверяет текущую ситуацию. Actions задают путь. Goal ждёт будущий результат до Deadline. Delivery Policy решает, как поступить с интерактивным действием, если пользователь offline.

Черновик можно менять и восстанавливать после перезагрузки. Опубликованная версия неизменяема. Каждый Scenario Run выполняет именно ту версию, с которой начался, а история объясняет, почему он стартовал и по какой ветке прошёл.
