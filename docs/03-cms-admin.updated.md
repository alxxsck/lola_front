# Техническое задание: CMS Lola

## 1. Назначение

CMS Lola - это SaaS-интерфейс для настройки ассистента, базы знаний, событий, элементов интерфейса, сценариев, визуального образа, голоса, стиля общения и аналитики.

Главная задача CMS - дать бизнес-команде и администраторам возможность управлять Lola без постоянного участия разработчиков после базовой интеграции.

## 2. Пользователи CMS

### 2.1. Owner

Управляет компанией, проектами, оплатой, доступами, API-ключами и глобальными настройками.

### 2.2. Admin

Настраивает ассистента, события, сценарии, базу знаний и интеграции.

### 2.3. Content Manager

Редактирует тексты, базу знаний, стиль общения, FAQ и контент сценариев.

### 2.4. Analyst

Смотрит аналитику, эффективность сценариев, события, конверсии и взаимодействия пользователей.

### 2.5. Developer

Получает ключи, документацию по интеграции, список событий, элементы интерфейса и технические параметры проекта.

## 3. Основные разделы CMS

- Dashboard;
- Projects;

- Knowledge Base;

- Publishing / Versions;
- Audit Log.

## 4. Dashboard

Dashboard должен показывать ключевое состояние проекта:

- статус интеграции;
- активность ассистента;
- количество диалогов;
- количество событий;
- активные сценарии;
- конверсию CTA;
- ошибки интеграции;
- последние изменения;
- рекомендации по настройке.

## 5. Настройки ассистента

Администратор должен иметь возможность настроить:

- имя ассистента;
- описание роли;
- язык по умолчанию;
- доступные языки;
- приветственное сообщение;
- позицию виджета;
- поведение при первом визите;
- поведение при повторном визите;
- режим текстового чата;
- режим голосового чата;
- ограничения по темам;
- правила передачи в поддержку.

## 6. Аватар и голос

CMS должна позволять:

- выбрать готового аватара;
- выбрать 3D-модель;
- просмотреть анимации;
- выбрать набор анимаций;
- выбрать голос;
- прослушать пример голоса;
- настроить скорость и эмоциональность речи, если это поддерживается;
- оставить заявку на кастомного персонажа;
- оставить заявку на кастомные анимации;
- оставить заявку на кастомный голос, если такая услуга предусмотрена.

Для кастомного персонажа необходимо хранить:

- описание персонажа;
- требования к стилю;
- список нужных анимаций;
- референсы;
- статус производства;
- комментарии команды.

## 7. База знаний

CMS должна позволять загружать и редактировать знания проекта.

Типы источников:

- ручные статьи;
- FAQ;
- Markdown;
- PDF;
- DOCX;
- CSV;
- ссылки на страницы;
- импорт из внешних систем.

Для каждого источника должны отображаться:

- название;
- тип;
- статус обработки;
- дата последнего обновления;
- версия;
- ошибки обработки;
- область применения;
- язык;
- автор изменения.

Администратор должен иметь возможность:

- добавить источник;
- удалить источник;
- отключить источник;
- обновить источник;
- запустить переиндексацию;
- протестировать вопрос по базе знаний;
- посмотреть, из каких источников был сформирован ответ.

## 8. Стиль общения

CMS должна позволять задавать характер ассистента.

Настройки:

- tone of voice;
- степень формальности;
- допустимый юмор;
- длина ответов;
- обращение к пользователю;
- слова и фразы, которые нужно использовать;
- запрещенные слова и фразы;
- темы, которых нужно избегать;
- правила отказа от ответа;
- правила эскалации в поддержку;
- языковые версии.

Пример:

```md
Lola общается дружелюбно и уверенно. Она объясняет коротко, не давит на пользователя, не обещает финансовый результат и не использует агрессивные продажи.
```

## 9. Карта интерфейса

Карта интерфейса - список элементов продукта, с которыми Lola может взаимодействовать.

Администратор должен иметь возможность добавить:

- кнопку;
- ссылку;
- форму;
- вкладку;
- модальное окно;
- страницу;
- блок интерфейса;
- custom action.

Для элемента задаются:

- название;
- системный код;
- тип;
- selector или data-атрибут;
- страница, где элемент находится;
- описание для админов;
- описание для AI;
- доступные действия;
- fallback, если элемент не найден.

Пример элемента:

```json
{
  "name": "Кнопка депозита",
  "code": "deposit_button",
  "type": "button",
  "selector": "[data-lola-action='deposit']",
  "page": "/account",
  "actions": ["highlight", "scroll_to", "click_hint"]
}
```

## 10. Страницы и модальные окна

CMS должна позволять описывать страницы и модальные окна продукта.

Страница:

- название;
- URL или route pattern;
- системный код;
- описание;
- доступна ли для прямого перехода;
- условия доступа.

Модальное окно:

- название;
- системный код;
- способ открытия;
- URL-параметр, если используется;
- frontend handler, если используется;
- описание;
- fallback-действие.

Пример:

```json
{
  "name": "Пополнение баланса",
  "code": "deposit_modal",
  "openMethod": "url_param",
  "param": "lola_modal",
  "value": "deposit"
}
```

## 11. События

CMS должна позволять создавать события, которые продукт клиента будет отправлять в Lola Backend.

Для события задаются:

- название;
- системный код;
- описание;
- источник;
- payload schema;
- активность;
- тестовый пример;
- связанные сценарии.

Пример:

```json
{
  "name": "Регистрация завершена",
  "code": "registration_completed",
  "source": "backend",
  "payloadSchema": {
    "language": "string",
    "country": "string",
    "registrationMethod": "string"
  }
}
```

После создания события CMS должна показывать разработчику:

- event code;
- пример API-запроса;
- обязательные поля;
- пример payload;
- статус последних вызовов.

## 12. Сценарии

Сценарии - ключевой раздел CMS. Он позволяет администратору настраивать поведение Lola без написания кода.

Сценарий должен иметь:

- название;
- системный код;
- статус;
- триггер;
- условия;
- список шагов;
- приоритет;
- ограничения по частоте;
- период активности;
- языковые версии;
- аналитику.

## 13. Конструктор сценариев

Конструктор должен позволять добавлять шаги:

- показать ассистента;
- скрыть ассистента;
- открыть чат;
- вывести текст;
- озвучить текст;
- проиграть анимацию;
- подсветить элемент;
- показать tooltip;
- показать CTA-кнопку;
- открыть страницу;
- открыть модальное окно;
- отправить событие;
- ожидать действие пользователя;
- проверить условие;
- завершить сценарий.

Пример сценария:

```md
Триггер: registration_completed

Шаг 1: показать Lola справа снизу.
Шаг 2: проиграть анимацию greeting.
Шаг 3: вывести текст: "Рада видеть вас внутри. Давайте сделаем следующий шаг."
Шаг 4: подсветить элемент deposit_button.
Шаг 5: показать CTA "Пополнить баланс".
Шаг 6: при клике открыть deposit_modal.
```

## 14. Условия сценариев

CMS должна поддерживать условия:

- пользователь новый или повторный;
- язык;
- страна;
- сегмент;
- страница;
- устройство;
- событие произошло впервые;
- событие произошло N раз;
- прошло N минут после события;
- пользователь уже видел сценарий;
- пользователь завершил другой сценарий;
- значение из payload события.

## 15. Ограничения частоты

Чтобы ассистент не мешал пользователю, сценарии должны иметь ограничения:

- не показывать чаще N раз в день;
- не показывать одному пользователю повторно;
- не показывать, если пользователь закрыл сценарий;
- не запускать одновременно с другим сценарием;
- учитывать приоритеты сценариев.

## 16. Публикация изменений

CMS должна поддерживать черновики и публикацию.

Процесс:

1. Администратор редактирует настройки.
2. Изменения сохраняются как draft.
3. Администратор проверяет сценарий в preview.
4. Изменения публикуются.
5. Frontend SDK и Backend начинают использовать опубликованную версию.

Желательно иметь возможность отката к предыдущей версии.

## 17. Интеграции

Раздел интеграций должен показывать:

- public project key;
- server API key;
- endpoint API;
- пример установки SDK;
- пример отправки события;
- список настроенных событий;
- список элементов интерфейса;
- статус последних запросов;
- ошибки интеграции.

## 18. Аналитика

CMS должна показывать:

- количество пользователей, видевших Lola;
- количество открытий ассистента;
- количество сообщений;
- количество голосовых сессий;
- клики по CTA;
- клики по подсвеченным элементам;
- запуски сценариев;
- завершения сценариев;
- отказы от сценариев;
- конверсию после сценариев;
- популярные вопросы;
- ошибки ассистента;
- события, требующие внимания.

## 19. Доступы и безопасность

CMS должна поддерживать:

- регистрацию и вход пользователей;
- роли;
- приглашения в проект;
- двухфакторную аутентификацию, если требуется;
- audit log;
- ограничение доступа к API-ключам;
- разделение компаний и проектов;
- безопасное хранение токенов.

## 20. Результат

CMS должна стать рабочим центром управления Lola: от настройки характера и базы знаний до создания сценариев, событий, CTA и аналитики. Пользователь CMS должен иметь возможность менять поведение ассистента быстро, прозрачно и без постоянных релизов frontend или backend продукта клиента.

## 21. Уточнение: границы первой версии CMS (MVP, 10.07.2026)

Разделы 1–20 описывают целевой продукт. Первая версия CMS имеет более узкие границы и не должна включать функции, которые ещё не поддержаны платформой или не нужны для проверки основного пользовательского пути.

### 21.1. Состав MVP

В первую версию входят:

- авторизация по рабочей почте с автоматическим определением единственного доступного проекта;
- работа только с заранее созданным проектом без project switcher;
- настройки текущего проекта: название, описание, языки, часовой пояс, имя и системная инструкция ассистента;
- CRUD кнопок и других элементов интерфейса;
- отдельное отображение и CRUD страниц;
- отдельное отображение и CRUD модальных окон;
- CRUD определений событий;
- конструктор payload события из полей: название, code, тип и обязательность;
- CRUD сценариев с trigger, ограничениями и упорядоченным списком действий;
- список пользователей, поиск, минимальные фильтры и карточка с timeline;
- список online/idle пользовательских сессий;
- ручная отправка в активную сессию текста, TTS-сообщения, CTA-кнопки или анимации.

В MVP не входят создание и переключение проектов, CRUD сотрудников CMS, приглашения, полноценный RBAC, 2FA, биллинг, Knowledge Base, Avatars & Voice management, Communication Style, полная аналитика, draft/publish/versioning, rollback, audit log и визуальные сценарии с ветвлениями.

Раздел 3.1 документа `04-integration-and-scenarios.md` для MVP следует читать как «Вход администратора в заранее созданный проект», а не как создание проекта в CMS.

### 21.2. Техническая реализация frontend

- Vue 3 SPA, TypeScript strict и Vite;
- Vue Router, Pinia и PrimeVue;
- FSD-like архитектура с практичным разделением на `app`, `pages`, `widgets`, `features`, `shared`;
- отдельный repository/API boundary, чтобы demo data не смешивались с production API;
- OpenAPI/Swagger как источник типов после того, как backend начнёт публиковать полные request/response schemas;
- обязательные loading, empty, error и destructive-confirm состояния;
- responsive layout с основной оптимизацией под desktop.

### 21.3. События

После ввода названия CMS предлагает уникальный lowercase code/slug, который администратор может изменить. Поля payload преобразуются в JSON Schema. Поддерживаемые в MVP типы: `string`, `number`, `integer`, `boolean`, `object`, `array`. Backend проверяет уникальность code и валидность schema при сохранении определения события.

### 21.4. Сценарии

Backend хранит сценарий как упорядоченный список, поэтому основной редактор MVP — последовательный action builder, а не свободный node graph. Визуальная схема может показывать trigger и порядок шагов, но не должна создавать ветвления, которые backend не умеет выполнять.

Минимальные actions: показать/скрыть ассистента, открыть/закрыть чат, вывести текст, проиграть анимацию, подсветить элемент, убрать подсветку, показать CTA, открыть страницу, открыть модальное окно, отправить track event и завершить сценарий. Перед сохранением frontend нормализует `position` от нуля.

CTA содержит label и целевое действие: открыть страницу, открыть модальное окно либо подсветить элемент. Страница, модальное окно и элемент выбираются из карты интерфейса проекта.

### 21.5. Пользователи, presence и ручные действия

Online означает наличие активного realtime-соединения с heartbeat не старше согласованного timeout. У одного пользователя может быть несколько сессий. Backend должен обновлять presence при connect, heartbeat и disconnect.

Карточка пользователя показывает объединённый timeline событий, запусков сценариев, сообщений, команд и ошибок.

Ручная команда всегда адресуется конкретной активной сессии и содержит `commandId`, `sessionId`, idempotency key и discriminated union payload. Статусы команды: `queued`, `delivered`, `failed`, `expired`. В MVP команда для offline-пользователя отклоняется.

Под `VOICE` в MVP понимается TTS: администратор вводит текст и при необходимости выбирает voice code. Загрузка или запись произвольного аудиофайла не входит в MVP.

### 21.6. Обязательные backend-контракты

Помимо существующих CRUD endpoint нужны:

- login, refresh, logout и `GET current CMS user/project`;
- paginated users с поиском и фильтрами;
- user detail и aggregated activity timeline;
- presence updates через SSE/WebSocket либо polling эффективнее текущего polling списка активных сессий;
- отдельный admin command endpoint нужен только если команды будут отправляться вне admin message;
- acknowledgement и delivery status ручной команды;
- каталоги допустимых animation и voice codes.

Корректный поток команды: `CMS → Lola Backend → realtime session channel → Frontend SDK → acknowledgement`. CMS не обращается к пользовательскому виджету напрямую.

### 21.7. Текущие интеграционные ограничения

Существующий backend покрывает CMS JWT auth, membership, настройки проекта, `UiElement`, определения событий, сценарии, пользователей, active presence, admin messages и operational logs. Swagger используется для надёжного code generation; оставшиеся пробелы — pagination, история диалогов, каталоги voice/animation и отдельный delivery/ACK журнал ручных команд.

Для production backend также не должен возвращать `serverKeyHash` в admin project response.

Текущий `ai-assistant` использует Bridge/AnyCable-контракты и не имеет универсального command dispatcher для `openPage`, `openModal`, highlight, CTA и admin commands. Его требуется привести к нормализованным командам Lola Backend и подтверждать результат выполнения каждой команды.
