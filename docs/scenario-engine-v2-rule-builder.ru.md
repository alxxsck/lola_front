# Scenario Engine V2: Rule Builder, проверка и preview

Документ подробно фиксирует frontend-срезы FE-V2-03, FE-V2-04 и FE-V2-05. Он остаётся справочником по Eligibility Rule Builder; актуальное завершение Audience и Delivery описано в `scenario-engine-v2-audience-unblock.ru.md`.

## Что видит пользователь

В Scenario Studio есть пять этапов: запуск, аудитория, условия, действия и доставка. Этот документ посвящён этапу "Условия". Audience и атомарная публикация Delivery уже реализованы отдельными typed-контрактами FE-V2-12/13. Старые `conditions` не смешиваются с Rule V2 и показываются отдельно как legacy-настройка.

Rule Builder не требует знания JSONPath или структуры DTO. Пользователь выбирает один из трёх источников:

- поле события запуска;
- историю событий пользователя;
- серию активных дней.

Группы поддерживают логику "все", "хотя бы одно" и отрицание. Условия можно перемещать, вкладывать в группы, редактировать и удалять. Справа всегда виден итог обычным языком: сколько условий собрано, нужна ли история и какой период будет запрошен.

Для быстрого старта есть два готовых примера: событие за последние семь дней и серия из трёх активных дней. После добавления это обычные условия — их можно открыть и поменять. Рядом есть короткая расшифровка логики групп, а в технических деталях видны текущие значения лимитов.

Редактор leaf открывается в Drawer. Изменения живут в локальном буфере до нажатия "Применить". "Отмена" ничего не меняет и возвращает фокус на кнопку, с которой пользователь открыл редактор. Если пользователь пытается уйти на другой этап или со страницы, незавершённые изменения нельзя потерять без подтверждения. На узких экранах Drawer занимает весь экран.

## Правила и ограничения

Доменная модель находится в `src/features/scenario-rules/model`. UI работает через команды и не меняет дерево напрямую.

Черновик хранит стабильные `nodeId` и `filterId`, но при сериализации они не попадают в backend DTO. Поддерживаются `all`, `any`, `not`, `eventField`, `eventAggregate` и `activityDayStreak`. Незнакомый kind сохраняется как opaque-узел без потери исходных данных; такой черновик можно открыть, но нельзя отправить как валидное правило.

Frontend заранее применяет ограничения backend: глубину дерева, число узлов и leaf, количество детей группы и aggregate-фильтров, длину массивов и строк, а также допустимые окна истории. В initial eligibility нельзя выбрать `sinceTrigger`. Событие запуска закреплено за точной ревизией из Scenario Authoring catalog, совпадения только по `eventCode` недостаточно.

Для денежных агрегатов UI сохраняет точное десятичное значение строкой и требует ровно один фильтр валюты с оператором `eq`. Оператор `in` всегда создаёт типизированный массив, а не одиночное значение. Значения и операторы берутся только из catalog capabilities. Недоступные после обновления каталога значения остаются видимыми в локальном буфере, чтобы пользователь мог осознанно заменить их или отменить редактирование. Чувствительные поля явно помечены, потому что preview может скрыть их фактическое значение.

## Проверка и preview

После изменения валидного черновика запускается debounce-проверка. Новый запрос отменяет предыдущий; поздний ответ не может перезаписать актуальное состояние. Локальные ошибки, semantic issues backend, сетевой сбой и неподдерживаемый формат ответа показаны раздельно.

Backend issue переводится из DTO path в `nodeId` и конкретное поле. Кнопка "Перейти" открывает нужный leaf и ставит фокус на проблемный control.

Preview работает только с реальным Event Log текущего проекта. Trigger event зафиксирован и не редактируется в фильтрах. Доступны external user ID, source, status, received/occurred interval и размер страницы. Cursor pagination хранит историю переходов, а retry повторяет именно тот запрос, который упал.

В backend отправляется только `eventLogId`, произвольный payload из браузера не принимается. Ответ проходит runtime-проверку и сверку topology explanation с текущим Rule. UI показывает результат, фактическое и ожидаемое значения, redaction, найденное количество, окно истории, cost, dependencies и warnings.

## Поведение при частичной недоступности backend

Scenario Authoring catalog и catalog действий загружаются независимо. Если Rule catalog недоступен, запуск и legacy-редактор остаются рабочими, а Rule Builder не строится на догадках. Если несовместим catalog действий, пользователь всё равно может настроить запуск и Rule; на этапе "Действия" будет отдельное предупреждение.

Backend пока не предоставляет source Rule для существующей ревизии и команду сохранения Rule-черновика. Поэтому FE-V2-03/04/05 держат Rule в текущей вкладке. При сохранении legacy-части Studio не уходит со страницы и прямо сообщает, что Rule V2 ещё не опубликован. Перезагрузку страницы нельзя считать сохранением Rule до следующих backend-срезов.

## Проверки

Поведение покрыто unit и component-тестами для команд AST, сериализации, mapping issues, естественных summary, Drawer, групп, focus recovery, request state machines, Event Log picker, explanation и интеграции Scenario Studio.

Ручной проход выполнен на живом Lola Backend: aggregate rule, backend validation, Event Log selection и preview explanation. Layout проверен при 1440, 1024, 768, 390 и 320 px, а также в landscape. На адаптивных размерах нет собственного горизонтального переполнения; этапы прокручиваются как компактная лента, а builder и preview идут последовательно по странице.
