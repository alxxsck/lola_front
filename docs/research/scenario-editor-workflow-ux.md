# UX-структура большого редактора сценариев

Дата исследования: 19 июля 2026 года.

## Задача

Определить, как упростить редактор сценария из пяти крупных разделов — «Запуск», «Аудитория», «Условия», «Действия» и «Доставка» — и оставить графовый холст только там, где пользователь действительно моделирует связи между действиями.

Источники ниже — официальная документация дизайн-систем IBM Carbon, PatternFly и Microsoft Fluent, а также официальная документация React Flow. Конкретные предложения для Lola являются выводами из этих рекомендаций, а не готовым шаблоном из одного источника.

## Решение в одном абзаце

Редактору нужен **полноэкранный пошаговый рабочий процесс с одной активной задачей**, а не три постоянно конкурирующие колонки. Слева остаётся компактная навигация по пяти разделам, в центре показывается только форма текущего раздела, а справа по запросу открывается контекстная справка. Графовый холст появляется только в разделе «Действия» и только после команды «Добавить первое действие»; при выборе узла справа открывается инспектор именно этого узла. В режиме просмотра формы превращаются в читаемое резюме, навигация и ссылки остаются рабочими, а изменение опубликованной версии начинается отдельным действием «Создать новый черновик».

## Почему это не должен быть жёсткий wizard

Carbon рекомендует progress indicator для линейного процесса из трёх и более шагов, особенно если данные шага нужно проверить перед переходом. Одновременно Carbon прямо не рекомендует его, когда разделы можно заполнять в любом порядке или их число меняется из-за условий. Интерактивный вариант разрешает переходить к шагам кликом и показывает состояния завершения и ошибки. ([Carbon: Progress indicator](https://carbondesignsystem.com/components/progress-indicator/usage/))

У сценария Lola пять устойчивых разделов, но администратору полезно возвращаться к ранее заполненным разделам и исправлять их без серии «Назад/Далее». Поэтому рекомендуется не модальный wizard с заблокированной последовательностью, а **in-page editor с интерактивной вертикальной навигацией по шагам**:

- порядок разделов остаётся понятным;
- доступный раздел открывается целиком в основной области;
- завершённость и ошибки видны в навигации;
- переход между разделами не зависит от права редактирования;
- публикация проверяет весь сценарий и ведёт к конкретному проблемному разделу.

PatternFly допускает in-page wizard, прямой переход к доступному шагу из sidebar и скрытие sidebar в раскрываемое меню на мобильном экране. ([PatternFly: Wizard](https://www.patternfly.org/components/wizard/design-guidelines/))

### Рекомендуемая структура

```text
┌────────────────────────────────────────────────────────────────────┐
│ ← Сценарии   Первый депозит     Черновик / Опубликованная версия   │
│                                      Проверить   Сохранить          │
├──────────────┬─────────────────────────────────────────────────────┤
│ ✓ Запуск     │ Заголовок текущего раздела                         │
│ 2 Аудитория  │ Короткое объяснение результата                     │
│ ! Условия    │                                                     │
│ ✓ Действия   │ Одна форма или одно рабочее представление          │
│ ✓ Доставка   │                                                     │
│              │                                                     │
│              │                         Справка ? (открывается)     │
└──────────────┴─────────────────────────────────────────────────────┘
```

Названия шагов должны быть короткими и описывать результат; Carbon предлагает формулу «глагол + объект» и отдельный helper text для необязательного или ошибочного шага. Для Lola устойчивые предметные названия уже короче и понятнее технических терминов. ([Carbon: Progress indicator](https://carbondesignsystem.com/components/progress-indicator/usage/))

## Где нужен холст, а где форма

React Flow определяет граф как узлы и рёбра внутри отдельного viewport с координатами, масштабом и перемещением. Его встроенные элементы управления относятся к масштабированию, вписыванию графа, выбору и редактированию узлов; `Panel` закрепляет элементы поверх этого viewport. Это специализированный инструмент для пространственного моделирования связей. ([React Flow: Terms and definitions](https://reactflow.dev/learn/concepts/terms-and-definitions), [React Flow: Built-in components](https://reactflow.dev/learn/concepts/built-in-components))

Отсюда следует разделение:

| Раздел | Основное представление | Почему |
| --- | --- | --- |
| Запуск | Обычная форма и карточка выбранного события | Пользователь задаёт свойства одного объекта, пространственных связей нет |
| Аудитория | Конструктор групп и условий в потоке страницы | Важнее читаемая логика «все/хотя бы одно», чем координаты блоков |
| Условия | Дерево групп и проверок | Нужны вложенность, ошибки и понятный порядок, но не свободное размещение |
| Действия | Графовый холст | Здесь пользователь строит последовательность, ветвления и переходы |
| Доставка | Форма политики выполнения | Это набор параметров одного решения |

Для «Действий» официальный пример React Flow использует библиотеку узлов рядом с pane и перенос узла в граф; PatternFly Topology помещает карту в отдельный viewer с toolbar, viewport controls и выдвижной панелью деталей. ([React Flow: Drag and drop](https://reactflow.dev/examples/interaction/drag-and-drop), [PatternFly: Topology view](https://v4-archive.patternfly.org/v4/documentation/react/components/topology/))

### Пустой раздел «Действия»

Не следует сразу показывать большое пустое поле с масштабированием. Сначала показывается компактное пустое состояние:

> **Добавьте первое действие**  
> Например, отправьте сообщение или дождитесь события. После добавления откроется схема сценария.  
> **Добавить действие**

После выбора типа создаётся первый узел и открывается холст. Это уменьшает количество незнакомых элементов управления до того, как они понадобились. Для touch-устройств React Flow отдельно предупреждает, что HTML Drag and Drop работает неполноценно, и предлагает pointer events; поэтому мобильный интерфейс не должен зависеть только от перетаскивания. ([React Flow: Drag and drop](https://reactflow.dev/examples/interaction/drag-and-drop))

## Правая панель: справка или инспектор, но не оба сразу

PatternFly рекомендует drawer внутри wizard, когда нужно показать дополнительный контекст, описание или обучающие материалы, не уводя пользователя с текущего шага. Такая панель открывается явной кнопкой и накладывается на содержимое, а не постоянно сжимает его. ([PatternFly: Wizard](https://www.patternfly.org/components/wizard/design-guidelines/))

Для Lola нужны два режима одной правой области:

1. **Контекстная справка** в разделах «Запуск», «Аудитория», «Условия» и «Доставка». По умолчанию закрыта. Открывается кнопкой «Справка по разделу» с иконкой вопроса. В ней нет обязательных настроек и основных действий.
2. **Инспектор выбранного узла** в «Действиях». Открывается после выбора узла, заголовок повторяет его название, закрытие возвращает весь размер холсту. На широком экране inspector может быть inline/resizable; на меньшем — overlay.

PatternFly различает inline drawer, который сохраняет основное содержимое видимым, и overlay drawer, который закрывается для возврата к перекрытой области. Для сложной primary-detail работы допускается splitter, если пользователь действительно нуждается в перераспределении места. ([PatternFly: Drawer](https://www.patternfly.org/components/drawer/design-guidelines/), [PatternFly: Primary-detail](https://v4-archive.patternfly.org/v4/demos/primary-detail/design-guidelines))

Постоянная правая колонка с общими объяснениями не подходит: она забирает ширину у формы и холста, хотя большую часть времени не является объектом работы. Техническую документацию следует открывать по запросу, а короткие обязательные подсказки оставлять непосредственно под полем.

## Прокрутка и кликабельность

Редактор должен занимать высоту окна и иметь предсказуемые области прокрутки:

```text
window/body             не прокручивается внутри editor route
editor header           закреплён сверху
step navigation         занимает высоту editor и остаётся видимой
main step content       единственная основная вертикальная прокрутка
context drawer          собственная прокрутка только когда открыт
canvas viewport         pan/zoom, а не прокрутка всей страницы
```

PatternFly Page поддерживает секции с собственным overflow scroll, заполнение доступной высоты, sticky header/footer и sidebar, который заполняет вертикальное пространство. Это подтверждает модель фиксированного shell с прокручиваемой основной секцией, а не случайный scroll на `body`. ([PatternFly: Page](https://www.patternfly.org/components/page/), [PatternFly: Page design guidelines](https://www.patternfly.org/components/page/design-guidelines/))

Практические требования к реализации:

- у цепочки `app → route → editor → content` должна быть определённая высота и `min-height: 0` у flex/grid-потомков;
- `overflow: hidden` задаётся editor shell, `overflow-y: auto` — main content и открытой drawer body;
- step navigation не должна лежать под невидимым overlay;
- слой canvas не должен перекрывать sidebar или inspector за пределами собственного контейнера;
- после перехода в другой раздел его основной заголовок получает программный фокус;
- кнопка из ошибки называется по назначению: «Открыть условия», а не «Перейти».

## Режим просмотра и версии

Carbon определяет read-only как активное информационное состояние: значение можно просмотреть и использовать, но нельзя изменить. Текст и значимые иконки сохраняют обычный контраст; фон и границы перестают имитировать интерактивность; chevron, календарь и другие signifier-иконки деакцентируются или убираются. Read-only содержимое остаётся доступным клавиатуре и screen reader, в отличие от disabled. ([Carbon: Read-only states](https://carbondesignsystem.com/patterns/read-only-states-pattern/))

Поэтому опубликованную или историческую версию нельзя показывать как большую форму из серых disabled inputs:

- в header — статус **«Опубликованная версия · только просмотр»** и короткая причина;
- содержимое — description list, карточки правил и читаемое резюме, без рамок input и chevron;
- переходы по разделам, раскрытие подробностей, копирование и просмотр графа остаются доступными;
- `Сохранить` и `Отмена` скрываются, потому что в этом состоянии нечего сохранять;
- отдельная основная кнопка **«Создать новый черновик»** начинает редактируемую версию, не меняя опубликованную;
- системный код причины (`SOURCE_SNAPSHOT_UNAVAILABLE`) не показывается в главном сообщении; он доступен внутри «Служебная информация» для поддержки.

Если исходный снимок старой версии нельзя восстановить, пользователю нужно сказать о последствии и следующем действии:

> **Эта версия сохранена без данных для редактора**  
> Сценарий продолжает работать как раньше, но изменить именно эту версию нельзя. Создайте новый черновик на основе последней доступной конфигурации.

Кнопка «Создать новый черновик» допустима только если backend действительно умеет построить совместимый черновик. Если такой операции нет, интерфейс должен предложить доступную альтернативу, а не неработающую кнопку. GOV.UK рекомендует не выдавать системную или permission-проблему за validation error, а объяснять ограничение и давать полезное следующее действие. ([GOV.UK: Error message](https://design-system.service.gov.uk/components/error-message/))

Carbon отдельно подчёркивает: disabled подходит для временной зависимости или prerequisites, а read-only — когда информацию нужно прочитать, но изменить нельзя. Значит, замок сам по себе недостаточен; рядом всегда нужен текстовый статус и причина. ([Carbon: Disabled states](https://carbondesignsystem.com/patterns/disabled-states/), [Carbon: Read-only states](https://carbondesignsystem.com/patterns/read-only-states-pattern/))

### Обратная совместимость на уровне UX

Интерфейс должен различать три состояния:

| Состояние | Что видит пользователь | Основное действие |
| --- | --- | --- |
| Черновик совместим с текущим редактором | Формы и холст доступны | Сохранить / Опубликовать |
| Опубликованная или историческая версия | Читаемое резюме, версия остаётся неизменной | Создать новый черновик |
| Исходник для редактора недоступен | Читаемое резюме того, что удалось восстановить, ясная причина | Создать совместимый черновик или вернуться к сценариям |

История версий должна быть отдельной областью просмотра. Выбор старой версии не переводит текущий editor в полусломанное состояние: route/header явно меняются на «Версия N · только просмотр», а возврат к черновику — отдельная кнопка.

## Адаптивность

Fluent описывает три базовые responsive-стратегии: reflow, show/hide второстепенного содержимого и re-architect с сохранением фокуса на главной информации и действиях. ([Fluent: Layout](https://fluent2.microsoft.design/layout))

Для Lola это даёт следующую схему:

| Ширина | Навигация | Основная область | Правая панель |
| --- | --- | --- | --- |
| Широкий desktop | Вертикальные шаги | Форма или canvas | Закрытая справка; inline inspector по выбору |
| Обычный desktop / tablet landscape | Компактные шаги | Одна колонка формы; canvas на всю оставшуюся ширину | Overlay drawer |
| Tablet portrait | Выпадающее меню текущего шага | Одна колонка | Полноширинный overlay |
| Mobile | Заголовок «Шаг N из 5» + список шагов | Формы в одну колонку; действия как упорядоченный список | Отдельный экран/полноэкранная панель |

PatternFly на мобильном скрывает step sidebar в выпадающее меню. Его sidebar также рассчитан на перестройку из side-by-side в stack. ([PatternFly: Wizard](https://www.patternfly.org/components/wizard/design-guidelines/), [PatternFly: Sidebar](https://v5-archive.patternfly.org/components/sidebar/design-guidelines))

Полноценный canvas на телефоне не должен быть единственным способом изменить граф. Минимально безопасный mobile fallback — список действий с командами «Добавить после», «Изменить», «Удалить» и явными переходами; визуальную схему можно оставить в режиме просмотра или открывать отдельным экраном.

## Предлагаемый сценарий работы

1. Администратор открывает сценарий и сразу видит его статус: черновик, опубликован или только для просмотра.
2. Слева видны пять разделов и состояние каждого. Нажатие всегда открывает выбранный раздел, если это не запрещено настоящим prerequisite.
3. В основной области находится только текущая задача. Ошибки показываются рядом с полем и в статусе раздела.
4. Справка открывается по кнопке и не отнимает ширину постоянно.
5. В разделе «Действия» пустое состояние предлагает добавить первый узел. После добавления появляется холст.
6. Выбор узла открывает inspector. Закрытие inspector не меняет выбор раздела и возвращает ширину холсту.
7. «Проверить сценарий» собирает ошибки по всем разделам. Каждая ошибка имеет конкретную кнопку: «Открыть аудиторию», «Открыть условия», «Открыть действие “…”».
8. Опубликованную версию можно читать и проверять, но её изменение всегда создаёт новый черновик.

## Проверяемые критерии готовности

- На разделах «Запуск», «Аудитория», «Условия» и «Доставка» нет canvas и его controls.
- До первого действия показывается одно понятное empty state, после добавления — граф.
- На ширине 1440 px основная форма не сжимается постоянной справочной колонкой.
- У main content и открытого inspector независимая вертикальная прокрутка; `body` во время работы editor не прокручивается.
- Все пять шагов кликабельны в режиме просмотра; навигационное действие не зависит от возможности сохранить.
- «Перейти» заменено на конкретные действия «Открыть …» и переводит фокус к заголовку нужного раздела.
- Read-only версия выглядит как читаемая информация, а не disabled form; текст проходит обычный контраст.
- Причина ограничения написана человеческим языком; технический код находится только в служебных подробностях.
- У опубликованной версии нет активных «Сохранить»/«Отмена»; есть рабочее действие для создания нового черновика или честная альтернатива.
- На tablet step navigation сворачивается, inspector становится overlay; на mobile доступен не drag-only способ управлять действиями.
- При увеличении текста до 200% нет горизонтального scroll всей страницы, перекрытия шагов и обрезанных кнопок.
