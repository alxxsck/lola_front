# Как настроить поля профиля пользователей

Поля профиля описывают, какие сведения о пользователе Lola получает от вашего продукта. Например:

- имя, которое можно подставить в сообщение;
- страна и валюта;
- уровень программы лояльности;
- количество депозитов;
- общая сумма депозитов в одной валюте;
- признак самоисключения.

Администратор настраивает не значения конкретного пользователя, а общие правила для всего Project: как называется поле, какой у него тип, обязательно ли его передавать и где Lola может его использовать. Сами значения передаёт сервер вашего продукта.

Здесь и дальше Project — это отдельный подключённый продукт в Lola со своими пользователями, полями, сегментами и сценариями.

Эта статья объясняет весь путь: от создания первого поля до публикации, изменения обязательности, безопасной замены и архивирования. Технические знания для основной настройки не нужны.

## Коротко: как устроены поля профиля

У каждого поля есть две стороны.

- Настройка поля говорит Lola, что означает значение и как с ним обращаться. Например: «Количество депозитов — целое число не меньше нуля».
- Значение поля относится к конкретному пользователю. Например: у пользователя `user-123` количество депозитов равно `7`.

Настройки действуют внутри одного Project. Поле из одного Project не появляется в другом.

Поля профиля показывают текущее состояние пользователя. Они отвечают на вопросы «что известно сейчас?» и «какое значение актуально сейчас?». История событий отвечает на другой вопрос: «что происходило раньше?»

Пример:

```text
Поле профиля: количество депозитов сейчас = 7
Событие: пользователь сделал депозит вчера в 18:42
```

Сегмент может использовать текущее количество депозитов. Условие сценария по истории событий может проверить, был ли депозит за последние семь дней. Это разные источники данных.

## Кто может менять поля

Владелец и администратор Project могут создавать, изменять, проверять и публиковать поля.

Пользователь с правом просмотра видит список и настройки, но не может их менять. Вверху страницы для него показано сообщение о режиме просмотра.

Некоторые изменения доступа к персональным и чувствительным данным требуют подтверждения именно владельца Project. Если администратор не может опубликовать такое изменение, это не ошибка кнопки: публикация ждёт владельца.

## Где находится раздел

Откройте пункт `Поля профиля` в основном меню.

В верхней части страницы находятся три действия.

- `Как передавать данные` открывает пошаговую инструкцию для подключения сервера продукта.
- `История изменений` показывает опубликованные версии, даты и причины публикации.
- `Добавить поле` открывает форму нового поля.

Ниже есть быстрые переходы:

- `Поля` возвращает к списку;
- `Передача данных` открывает инструкцию подключения;
- `Качество данных` прокручивает страницу к статистике;
- `История` открывает историю опубликованных версий.

## Черновик и опубликованная версия

Изменения не начинают действовать сразу. Сначала они попадают в общий черновик Project.

На странице одновременно показаны два состояния:

- `Опубликованная версия` — структура, которую Lola использует сейчас;
- `Текущий черновик` — подготовленные изменения, которые ещё не действуют.

Один черновик может содержать несколько изменений: новое поле, новое название другого поля и вывод третьего поля из использования. Все они публикуются вместе как следующая версия структуры.

Это защищает работающую интеграцию от незаконченной настройки. Можно спокойно сохранить черновик, проверить предупреждения и только потом включить новую структуру.

### Что означает версия

При каждой публикации Lola создаёт новую неизменяемую версию всего набора полей.

Например:

```text
Версия 3: displayName, depositCount
Версия 4: displayName, depositCount, lifetimeDepositsEur
```

Версия 3 не переписывается и остаётся в истории. Версия 4 становится текущей.

Старую версию можно временно принимать во время переходного периода, чтобы команда продукта успела обновить интеграцию. Этот период задаётся при публикации и может составлять от 0 до 30 дней. По умолчанию интерфейс предлагает 7 дней.

## Как создать поле

1. Откройте `Поля профиля`.
2. Нажмите `Добавить поле`.
3. Заполните раздел `Что хранится в поле`.
4. Укажите, где Lola может использовать значение.
5. При необходимости откройте `Расширенные настройки`.
6. Нажмите `Добавить в черновик`.
7. Вернитесь к списку и выполните три шага внизу страницы: `Сохранить`, `Проверить`, `Опубликовать`.

Добавление в черновик ещё ничего не меняет для работающих пользователей и сценариев.

## Основные настройки поля

### Название поля

`Название поля` видят администраторы. Пишите его так, чтобы смысл был понятен без знания кода продукта.

Хорошие названия:

- `Отображаемое имя`;
- `Количество депозитов`;
- `Общая сумма депозитов в EUR`;
- `Уровень лояльности`.

Слишком общие названия вроде `Статус`, `Сумма` или `Тип` быстро становятся непонятными.

Название можно исправить в следующей версии. Это не меняет техническую личность поля.

### Ключ для передачи данных

`Ключ для передачи данных` использует сервер вашего продукта. Например: `displayName`, `depositCount` или `lifetimeDepositsEur`.

Ключ:

- начинается с латинской буквы;
- не содержит пробелов;
- после первой публикации не меняется;
- не может быть повторно использован для другого смысла, даже после архивирования.

Если ключ выбран неправильно, удалите поле из черновика до первой публикации. Если поле уже опубликовано, создайте новое поле с новым ключом и перенесите интеграцию.

### Тип данных

Тип определяет, какие значения Lola примет и какие проверки будут доступны в сегментах.

| Тип в интерфейсе   | Для чего подходит                              | Пример                 |
| ------------------ | ---------------------------------------------- | ---------------------- |
| `Текст`            | имя, статус, уровень лояльности                | `gold`                 |
| `Да или нет`       | признак с двумя состояниями                    | `да`                   |
| `Целое число`      | количество депозитов, заказов или входов       | `7`                    |
| `Десятичное число` | деньги, баланс, рейтинг                        | `1250.50`              |
| `Дата`             | день рождения или дата регистрации без времени | `2026-07-19`           |
| `Дата и время`     | точный момент с часовым поясом                 | `2026-07-19T08:30:00Z` |
| `Страна`           | двухбуквенный код страны                       | `ES`                   |
| `Валюта`           | трёхбуквенный код валюты                       | `EUR`                  |

Тип после первой публикации не меняется. Поле `depositCount` нельзя превратить из текста в число. Для этого создают новое поле, переводят продукт и сценарии, затем выводят старое из использования.

### Обязательно ли передавать поле

Есть три режима.

- `Можно не передавать` — отсутствие значения считается допустимым.
- `Предупреждать, если значения нет` — профиль принимается, но Lola отмечает проблему качества данных.
- `Не принимать профиль без значения` — обновление профиля без этого поля отклоняется.

Последний режим не блокирует пользователю вход или создание сессии. Lola продолжит работать с последним корректным профилем, но новое неполное обновление не заменит его.

Безопасный порядок для нового обязательного поля:

1. Создайте поле как необязательное.
2. Попросите команду продукта начать передавать его.
3. Проверьте заполненность в `Качество данных`.
4. Включите режим предупреждения.
5. Убедитесь, что старые интеграции обновлены.
6. Только после этого включите строгую обязательность.

При переходе к строгой обязательности публикация попросит подтверждение готовности интеграции. Это защита от ситуации, когда один переключатель внезапно отклоняет все новые профили.

### Описание

В описании ответьте на два вопроса:

- что означает значение;
- откуда оно берётся.

Пример:

```text
Количество успешно завершённых депозитов за всё время.
Рассчитывается платёжным сервером продукта.
```

### Для чего нужно это поле

Назначение объясняет, зачем Lola получает эти данные. Оно обязательно для персональных и чувствительных данных, а также когда значение разрешено использовать в других разделах.

Хорошая формулировка:

```text
Собирать аудитории по уровню лояльности и подставлять уровень в сообщения.
```

Формулировка `Нужно для работы` ничего не объясняет и не помогает при проверке доступа.

### Допустимые значения

Список ограничивает поле заранее известными вариантами. Каждое значение вводится с новой строки.

Для уровня лояльности:

```text
basic
silver
gold
vip
```

После публикации продукт не сможет передать другое значение, например `premium`, пока оно не добавлено в новую версию настроек.

Добавление нового допустимого значения расширяет контракт. Удаление значения или сужение диапазона опаснее: старые профили и интеграции могут продолжать его отправлять. Перед таким изменением проверьте данные и подготовьте план перехода.

## Где можно использовать поле

Каждый переключатель открывает отдельный способ чтения данных. Не включайте доступ «на всякий случай».

### Показывать администраторам

Поле видно в карточке пользователя. Подходит для данных, которые нужны поддержке или операционной команде.

### Использовать в сегментах

Поле появляется в конструкторе аудиторий и сегментов. Включайте, если по значению действительно будут собирать группы пользователей.

Сам переключатель не создаёт сегмент и не индексирует все данные автоматически. Он только разрешает использовать поле в правилах.

### Использовать в шаблонах

Значение можно подставлять в сообщения. Например: `Здравствуйте, {{ displayName }}`.

Не открывайте шаблонам внутренние или чувствительные значения, если они не должны появляться в тексте для пользователя.

### Разрешить ИИ использовать значение

Функции Lola с ИИ смогут получить это поле в своём ограниченном контексте.

Не включайте этот доступ для паролей, токенов, секретных ответов и других секретов. Такие данные вообще не должны становиться полями профиля Lola.

Расширение доступа к персональным или чувствительным данным требует отдельного подтверждения при публикации.

### Показывать на сайте

Значение получает код страницы в браузере пользователя. Браузер нельзя считать защищённым местом для секретов или внутренних финансовых данных.

### Разрешить экспорт

Поле может попасть в выгрузки. Учитывайте правила доступа и хранения персональных данных за пределами Lola.

## Расширенные настройки

### Категория данных

- `Служебные данные` — внутренние сведения без специальной категории.
- `Персональные данные` — сведения, связанные с конкретным человеком.
- `Чувствительные данные` — данные, для которых нужна особенно строгая проверка доступа.

Категория не выдаёт доступ автоматически. Она помогает Lola применить дополнительные ограничения.

### Поиск и фильтрация

- `Не использовать для поиска` — поле не предназначено для фильтра в списке пользователей.
- `Искать по точному значению` — подходит для страны, валюты, статуса или уровня.
- `Фильтровать и сортировать` — подходит для чисел и дат, когда нужны сравнения «больше», «меньше» и сортировка.

На странице `Профили пользователей` поиск по полю является разовым фильтром списка. Он не создаёт сохраняемую аудиторию. Для переиспользования группы в сценариях создайте сегмент.

### Системное назначение

Системное назначение сообщает Lola, что поле играет одну из специальных ролей:

- имя пользователя;
- электронная почта;
- страна;
- валюта;
- язык контента.

При создании поля сначала выберите его назначение в верхнем блоке формы. Для
системной роли Lola сразу предложит название, ключ и совместимый тип данных, а
для языка контента также откроет настройку доступных языков. Предложенные
название и ключ можно изменить: при последующей смене роли введённый вручную
текст не стирается.

У каждой роли может быть только одно активное поле в Project. Не назначайте роль просто потому, что название похоже. Например, `Валюта последнего депозита` не является основной валютой пользователя.

### Состояние поля

- `Активно` — поле используется в текущей настройке.
- `Выводится из использования` — поле ещё принимается, но его нужно заменить и перестать использовать в новых настройках.

Вывод из использования не равен удалению. Это период миграции.

Для выводимого поля выберите режим `Можно не передавать`. Старый продукт ещё может присылать значение, но новый профиль не должен отклоняться из-за того, что поле уже убрали из интеграции.

После публикации состояния `Выводится из использования` вернуть поле в активное состояние нельзя. Если планы изменились, создайте подходящее поле-замену вместо попытки оживить старую настройку.

В блоке `Как вывести поле из использования` есть две настройки.

- `ID поля-замены` — постоянный идентификатор уже созданного нового поля, а не его название или ключ. В текущем интерфейсе его можно взять из адреса страницы нового опубликованного поля после `/profile-fields/`. Если вы не уверены, не угадывайте значение: попросите владельца Project или разработчика помочь с точным ID.
- `Не использовать после` — дата и время окончания перехода в формате вроде `2026-12-31T00:00:00Z`.

### Ограничения значения

Для текста можно задать минимальную и максимальную длину.

Для чисел можно задать минимум и максимум. Для десятичного числа также задаются:

- `Всего цифр` — общее количество цифр;
- `Знаков после запятой` — количество цифр в дробной части.

Например, значение `1250.50` содержит шесть цифр, из них две после точки.

Не сужайте ограничения, пока продукт продолжает передавать старые значения. Одно неподходящее поле приводит к отклонению всего нового снимка профиля, а не только одной ячейки.

### Лимиты Project

Проверка черновика также следит за общими лимитами.

- В Project может быть не больше 50 неархивированных полей. После 20 полей Lola уже показывает предупреждение, чтобы структура не разрасталась без необходимости.
- Для поиска и сортировки можно подготовить не больше 10 полей.
- Функциям с ИИ можно открыть не больше 10 полей.
- В списке допустимых значений может быть от 1 до 100 уникальных вариантов.

Если проверка сообщает о превышении лимита, не пытайтесь обойти её дублирующим полем. Уберите ненужный доступ, завершите архивирование старых полей или пересмотрите структуру вместе с владельцем Project.

## Полный пример: поля для игрового продукта

Ниже не обязательный шаблон, а пример того, как размышлять о настройке.

### Отображаемое имя

```text
Название: Отображаемое имя
Ключ: displayName
Тип: Текст
Обязательность: Предупреждать, если значения нет
Назначение: Обращаться к пользователю по имени в интерфейсе и сообщениях
Доступ: администраторы, шаблоны
Системное назначение: Имя пользователя
```

Не делайте имя строго обязательным в первый день. Сначала посмотрите, у какой доли профилей оно заполнено.

### Количество депозитов

```text
Название: Количество депозитов
Ключ: depositCount
Тип: Целое число
Обязательность: Можно не передавать
Минимальное значение: 0
Назначение: Собирать аудитории по количеству успешных депозитов
Доступ: администраторы, сегменты
Поиск: Фильтровать и сортировать
```

Это текущее число, которое рассчитывает продукт и передаёт Lola. Сегмент не подсчитывает события самостоятельно.

### Общая сумма депозитов в EUR

```text
Название: Общая сумма депозитов в EUR
Ключ: lifetimeDepositsEur
Тип: Десятичное число
Обязательность: Можно не передавать
Минимальное значение: 0
Знаков после запятой: 2
Назначение: Собирать аудитории по общей сумме успешных депозитов в EUR
Доступ: администраторы, сегменты
Категория: Чувствительные данные
Поиск: Фильтровать и сортировать
```

Не храните в одном числе сумму разных валют. Используйте сумму в одной общей валюте продукта либо отдельные поля с явной валютой в названии и описании.

### Уровень лояльности

```text
Название: Уровень лояльности
Ключ: loyaltyTier
Тип: Текст
Допустимые значения: basic, silver, gold, vip
Назначение: Собирать аудитории и выбирать подходящий вариант сообщения
Доступ: администраторы, сегменты, шаблоны
Поиск: Искать по точному значению
```

### Признак самоисключения

```text
Название: Самоисключение активно
Ключ: selfExcluded
Тип: Да или нет
Назначение: Исключать пользователя из неподходящих аудиторий
Доступ: только необходимые административные и сегментационные проверки
Категория: Чувствительные данные
```

Не открывайте это значение сайту, шаблонам, экспорту или ИИ без отдельной подтверждённой причины.

## Как сохранить, проверить и опубликовать изменения

Внизу страницы есть три кнопки. Их нужно нажимать по порядку.

### 1. Сохранить

Кнопка записывает текущий черновик на сервер. Изменения всё ещё не действуют.

Если другой администратор уже сохранил свою версию, Lola не перезапишет её молча. Появится сообщение `Нужна ручная сверка черновиков`.

- `Скопировать свои изменения` сохраняет локальный вариант в буфер обмена.
- `Открыть сохранённую версию` загружает версию другого администратора.

После этого изменения нужно сравнить и перенести вручную.

### 2. Проверить

Lola проверяет ключи, типы, ограничения, обязательность, доступ к данным и совместимость с опубликованной версией.

Ошибки блокируют публикацию. Кнопка `Исправить поле` или `Проверить поле` открывает нужную настройку.

Проверку нужно запускать заново после каждого изменения черновика.

### 3. Опубликовать

Кнопка становится доступна после сохранения и успешной проверки.

В окне публикации заполните:

- `Причина изменения` — коротко опишите, что изменилось и зачем, например `Добавили поля для VIP-сегментации`; этот текст появится в истории публикаций;
- `Переходный период, дней` — сколько дней Lola будет принимать предыдущую версию полей, пока разработчики обновляют интеграции; указывайте `0`, только когда все источники уже готовы;
- `План перехода для несовместимых изменений` — какие системы и в каком порядке обновит команда, если старые запросы перестанут подходить новой структуре;
- `Номер задачи о готовности интеграции` — задача или проверка, где разработчики подтвердили готовность передавать новые обязательные поля;
- подтверждение проверки доступа к персональным и чувствительным данным — требуется для изменений безопасности.

После публикации создаётся новая версия. Ранее сохранённые профили и старые версии не переписываются.

## Как изменить опубликованное поле

1. Найдите поле в списке.
2. Нажмите `Изменить`.
3. Измените разрешённые настройки.
4. Нажмите `Сохранить изменения`.
5. На общей странице нажмите `Сохранить`, `Проверить`, `Опубликовать`.

Название, описание, назначение, режим обязательности, доступ и некоторые ограничения можно изменить через новую версию.

Ключ и тип заблокированы после создания. Это сделано специально: сценарии, сегменты и сервер продукта должны понимать поле одинаково.

## Что делать, если нужно поменять смысл или тип поля

Не используйте старый ключ для нового смысла.

Например, поле `depositAmount` раньше означало сумму последнего депозита, а теперь должно означать сумму всех депозитов. Это другое поле, даже если оба значения являются числами.

Безопасный порядок:

1. Создайте новое поле с точным названием и новым ключом, например `lifetimeDepositsEur`.
2. Оставьте его необязательным и опубликуйте.
3. Обновите сервер продукта, чтобы он передавал старое и новое поле.
4. Проверьте заполненность нового поля.
5. Переведите сегменты, сценарии и шаблоны на новое поле.
6. У старого поля выберите `Выводится из использования`.
7. Укажите поле-замену и дату, после которой старое поле не нужно использовать.
8. Когда зависимостей не останется, архивируйте старое поле.

## Как удалить поле

Ответ зависит от того, публиковалось ли оно раньше.

### Новое поле только в черновике

У такого поля есть кнопка с корзиной. Она удаляет поле из черновика. Поскольку поле ещё не действовало, история и интеграции от него не зависят.

### Опубликованное поле

Кнопки обычного удаления нет. Опубликованное поле сначала выводят из использования, затем архивируют.

1. Нажмите `Где используется`.
2. Проверьте связанные сегменты, сценарии, шаблоны и другие разделы.
3. Если есть зависимости, переведите их на поле-замену.
4. Измените состояние поля на `Выводится из использования`, укажите замену и дату окончания использования.
5. Сохраните, проверьте и опубликуйте эту версию.
6. Снова откройте `Где используется`.
7. Когда блокирующих зависимостей нет, нажмите `Добавить архивирование в черновик`.
8. Ещё раз сохраните, проверьте и опубликуйте.

Архивирование:

- убирает поле из текущей структуры и новых профилей;
- запускает очистку этого значения из текущих профилей;
- сохраняет описание поля, опубликованные версии и другую историческую информацию;
- не освобождает ключ для повторного использования;
- не переписывает старые опубликованные версии.

Архивное поле нельзя снова сделать активным. Перед подтверждением убедитесь, что миграция действительно закончена.

## Что показывает «Где используется»

Окно помогает понять последствия изменения или архивирования.

Если зависимостей нет, Lola показывает, что поле не используется в активных сегментах, шаблонах или других настройках.

Если зависимости есть, они перечисляются с типом и состоянием. Архивирование может быть заблокировано, пока активные настройки продолжают ссылаться на поле.

Не пытайтесь обходить блокировку созданием поля с тем же ключом. Ключ остаётся закреплён за старым полем навсегда.

## Качество поступающих данных

Раздел `Качество данных` помогает проверить, что продукт действительно отправляет новую структуру.

Выберите период: 24 часа, 7 или 30 дней. На странице показаны:

- сколько обновлений получено;
- сколько сессий имели профиль и сколько были без него;
- сколько повторных запросов конфликтовало;
- когда пришло последнее успешное обновление;
- сколько запросов ожидают обработки;
- возраст сохранённых профилей;
- заполненность отдельных полей, если она доступна;
- интеграции, которые продолжают использовать старую версию.

Перед строгой обязательностью ориентируйтесь не на один тестовый профиль, а на реальную заполненность за подходящий период.

Ссылка `Посмотреть исходные данные для разработчика` нужна для диагностики. В повседневной работе администратору достаточно человекочитаемых показателей выше.

## Как передать значения после публикации

Откройте `Как передавать данные`.

Страница ведёт по четырём шагам:

1. убедиться, что поля опубликованы;
2. выбрать способ передачи;
3. отправить тестовый профиль;
4. проверить результат в карточке пользователя.

Есть два способа передачи.

- `Обновлять профиль при изменении` подходит для синхронизации из CRM или сервера продукта.
- `Передавать профиль при запуске сессии` подходит, если актуальные данные собираются перед общением с пользователем.

Оба способа отправляют полный профиль. Новый корректный снимок заменяет предыдущий. Поэтому сервер продукта должен передавать все актуальные поля, а не только изменившееся значение.

Технический пример можно передать разработчику. Серверный токен нельзя помещать в браузер.

## Как проверить значение у пользователя

1. Откройте `Профили пользователей`.
2. Найдите пользователя по внешнему ID или примените фильтр по индексируемому полю.
3. Откройте карточку пользователя.
4. Проверьте версию профиля, версию полей, время актуальности и состояние обновления.

Если последнее обновление было отклонено, Lola сохраняет предыдущий корректный профиль и показывает причины отдельно.

Переключатель `Архивные` показывает поля, которые уже убраны из текущей структуры, но сохранились в истории. Переключатель `Ключи` показывает технические ключи.

## Данные для ИИ

Владелец Project может открыть карточку `Данные для ИИ` на странице полей.

Она показывает только опубликованные поля, для которых включено разрешение `Разрешить ИИ использовать значение`. Предпросмотр содержит вымышленные значения и помогает проверить состав контекста без раскрытия данных реального пользователя.

Изменение формата или размера предпросмотра не меняет политику доступа. Чтобы добавить или убрать поле, измените его настройку, проверьте и опубликуйте новую версию.

## Частые вопросы

### Почему поле не появилось в сегментах?

Проверьте три вещи:

1. включён ли переключатель `Использовать в сегментах`;
2. опубликована ли версия с этим изменением;
3. поддерживает ли тип поля сравнения, которые вы хотите использовать.

### Почему изменения сохранены, но не действуют?

Сохранение записывает черновик. После него нужно выполнить проверку и публикацию.

### Почему нельзя поменять ключ или тип?

На них уже могут ссылаться сервер продукта, сегменты и сценарии. Создайте поле-замену с новым ключом.

### Можно ли сделать поле обязательным сразу?

Технически переход возможен через публикацию, но безопаснее пройти режим предупреждения и проверить заполненность. Строгая обязательность отклоняет весь неполный снимок профиля.

### Что произойдёт, если продукт пришлёт неправильное значение?

Новый снимок будет отклонён целиком. Последний корректный профиль останется доступен. Сессия пользователя при этом не должна блокироваться из-за ошибки профиля.

### Можно ли очистить одно значение, передав пустое поле?

Текущая модель принимает полный снимок профиля. Пропущенное необязательное поле означает, что в новом снимке значения нет. Отдельного частичного редактирования одной ячейки администратором нет.

### Можно ли хранить пароль или токен?

Нет. Секреты, учётные данные и ответы безопасности не должны становиться полями профиля Lola.

### Чем поле отличается от события?

Поле хранит текущее состояние. Событие фиксирует произошедший факт и остаётся в истории.

### Чем фильтр пользователей отличается от сегмента?

Фильтр помогает один раз найти пользователей в списке. Сегмент сохраняет правило, имеет версии и может использоваться в нескольких сценариях.

### Что делать, если два администратора редактировали черновик одновременно?

Lola не выбирает победителя автоматически. Скопируйте свои изменения, откройте сохранённую серверную версию и вручную перенесите нужные правки.

## Проверочный список перед публикацией

- Название и описание понятны человеку, который не участвовал в настройке.
- Ключ отражает один устойчивый смысл.
- Тип соответствует реальным данным продукта.
- Денежная сумма имеет понятную валюту.
- Обязательность соответствует фактической заполненности.
- Допустимые значения уже поддерживаются продуктом.
- Доступ администраторам, сегментам, шаблонам, ИИ, сайту и экспорту включён только там, где нужен.
- Персональные и чувствительные данные имеют понятное назначение.
- Для несовместимого изменения подготовлен план перехода.
- Черновик сохранён и успешно проверен.
- В причине публикации написано, что именно меняется.

После публикации откройте `Качество данных` и убедитесь, что продукт начал использовать новую версию без роста ошибок.
